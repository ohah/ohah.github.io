# 3-2. 스크립팅 엔진과 트래픽 조작

## Deno Core 임베딩: V8을 Rust에 넣다

프록시에 TypeScript 스크립팅을 넣으려면 JavaScript 런타임이 필요하다. 선택지는 세 가지였다:

| 옵션 | 장점 | 단점 |
|------|------|------|
| QuickJS | 가볍다 (수백 KB) | V8 대비 성능 1/10 |
| Node.js 임베드 | 생태계 풍부 | 무겁고 Rust 연동 복잡 |
| **Deno Core** | V8 성능 + Rust 네이티브 | 바이너리 크기 증가 |

Deno Core를 선택했다. 이유:
1. Deno 자체가 Rust + V8 조합으로 검증됨
2. `deno_core` 크레이트로 V8 런타임을 Rust에서 직접 제어 가능
3. 커스텀 API를 Rust op으로 등록하면 JavaScript에서 호출 가능

### 아키텍처

```
[TypeScript 스크립트]
        ↓ oxc (트랜스파일)
[JavaScript]
        ↓ deno_core
[V8 엔진]
        ↓ Rust ops
[프록시 코어 API]
```

사용자가 TypeScript를 작성하면:
1. **oxc**로 TypeScript → JavaScript 변환 (타입 제거)
2. **deno_core**가 V8 런타임에 로드
3. 스크립트에서 `cheolsu.onRequest()` 등을 호출하면 Rust op이 콜백 등록
4. 실제 트래픽이 들어오면 등록된 콜백 실행

### 트러블슈팅: oxc 트랜스파일 파이프라인 <Badge type="danger">삽질</Badge>

처음에는 `tsc`를 외부 프로세스로 호출해서 TypeScript를 변환했다. 근데 문제가 많았다:

1. **속도** — `tsc`는 느리다. 파일 하나 변환에 수백 ms
2. **의존성** — Node.js가 시스템에 설치되어 있어야 함
3. **프로세스 관리** — 외부 프로세스 생명주기 관리가 번거롭다

oxc로 전환하면서 전부 해결됐다:
- Rust 네이티브 호출이라 sub-ms 수준
- 외부 의존성 없음
- `oxc_parser` → `oxc_transformer` → `oxc_codegen` 파이프라인이 깔끔

단, oxc는 타입 체킹을 안 한다. 순수하게 타입 구문만 제거한다. 타입 에러는 에디터(Monaco)에서 잡아주는 걸로 충분하다고 판단했다.

### 트러블슈팅: V8 스냅샷과 메모리 <Badge type="warning">성능</Badge>

Deno Core는 V8 스냅샷을 지원한다. 런타임 초기화 상태를 스냅샷으로 저장해두면 다음 실행 시 빠르게 복원할 수 있다.

근데 스크립팅 모듈의 바이너리 크기가 예상보다 컸다. V8 자체가 수십 MB이고, Deno Core가 추가로 들어가니까.

릴리스 빌드에서 LTO + strip + `opt-level = "s"`로 최대한 줄였지만, 스크립팅을 켜면 바이너리가 확 커진다. 이건 V8의 태생적 한계라 어쩔 수 없다.

## 세 가지 훅: Request, Response, WebSocket

스크립팅의 핵심은 세 가지 훅이다:

### 1. `cheolsu.onRequest()` — 요청 가로채기

```typescript
cheolsu.onRequest((request) => {
  // request: { method, url, headers, body? }

  // 그대로 통과
  return { action: "forward" };

  // 수정해서 전달
  return { action: "modify", request: { ...request, headers: { ...request.headers, "X-Custom": "value" } } };

  // 서버에 보내지 않고 직접 응답
  return { action: "respond", response: { status: 200, headers: {}, body: "mocked" } };
});
```

세 가지 반환 타입이 핵심이다:
- `forward` — 원본 요청 그대로 서버에 전달
- `modify` — 수정된 요청을 서버에 전달
- `respond` — 서버에 보내지 않고 프록시에서 직접 응답 (목업)

### 2. `cheolsu.onResponse()` — 응답 수정

```typescript
cheolsu.onResponse((request, response) => {
  // request와 response 둘 다 접근 가능
  if (request.url.includes("/api/user") && response.status === 200) {
    const body = JSON.parse(response.body || "{}");
    body.debugInfo = { proxied: true, timestamp: Date.now() };
    return {
      action: "modify",
      response: { ...response, body: JSON.stringify(body) }
    };
  }
  return { action: "forward" };
});
```

### 3. `cheolsu.onWebSocketMessage()` — WebSocket 메시지 조작

```typescript
cheolsu.onWebSocketMessage((message) => {
  // message: { connection_id, url, direction, payload, is_binary }

  // 메시지 드롭 (무시)
  if (message.payload.includes("heartbeat")) {
    return { action: "drop" };
  }

  // 메시지 수정
  return { action: "forward" };
});
```

`drop` 액션이 독특하다. 특정 메시지를 완전히 무시할 수 있다. 노이즈가 많은 heartbeat 같은 메시지를 필터링할 때 유용하다.

### 트러블슈팅: 비동기 훅과 V8 이벤트 루프 <Badge type="danger">삽질</Badge>

초기에 동기 훅만 지원했다. 근데 실제로 쓰다 보니 비동기가 필요한 경우가 많았다. 예를 들어 요청을 가로채서 다른 API의 결과를 기반으로 응답을 조작하고 싶은 경우.

Deno Core의 V8 이벤트 루프와 Tokio의 비동기 런타임을 연결하는 게 까다로웠다. V8은 자체 이벤트 루프가 있고, Tokio도 자체 이벤트 루프가 있다. 둘을 동기화하지 않으면 데드락이 걸린다.

해결: `deno_core::JsRuntime`의 `run_event_loop`를 Tokio 태스크 안에서 실행하되, 훅 실행은 별도 채널로 요청/응답을 주고받는 구조로 만들었다. 프록시 코어 → (채널) → V8 런타임 → (채널) → 프록시 코어.

### 자동 리로드

파일 기반 스크립트는 변경을 감지해서 자동으로 리로드한다. 500ms 디바운스를 적용해서 저장할 때마다 스크립트가 재실행된다.

에디터에서 스크립트를 수정하면 → 파일 변경 감지 → 500ms 대기 → V8 런타임 재생성 → 새 스크립트 로드. 개발 중에 매번 프록시를 재시작할 필요가 없다.

## Cheolsu-Query: 전용 쿼리 언어

트래픽이 많아지면 필터링이 필수다. 단순 텍스트 검색으로는 부족해서 전용 쿼리 언어를 만들었다.

### 문법

```
method="GET" and status="2xx" and url|="api"
(method="POST" or method="PUT") and status="5xx"
url|~"localhost" or url|~"127.0.0.1"
```

연산자:
| 연산자 | 의미 |
|--------|------|
| `=` | 정확 매칭 |
| `\|=` | 포함 (대소문자 구분) |
| `\|~` | 포함 (대소문자 무시) |
| `!=` | 불일치 |
| `!~` | 미포함 |

논리 연산자: `and`, `or`, `()` 그룹

### 트러블슈팅: 쿼리 파서 구현 <Badge type="warning">설계</Badge>

처음에는 정규식으로 쿼리를 파싱했다. 간단한 쿼리는 됐지만 `()` 그룹과 `or` 조합이 들어가니 한계에 부딪혔다.

결국 재귀 하강 파서(recursive descent parser)를 직접 구현했다:
1. Lexer — 토큰화 (`method`, `=`, `"GET"`, `and`, `(`, `)`)
2. Parser — AST 생성 (And/Or/Not 노드)
3. Evaluator — AST를 트래픽 데이터에 대해 평가

Monaco Editor에 자동 완성도 연결했다. `method=`를 치면 `"GET"`, `"POST"` 등이 제안되고, `status=`를 치면 `"2xx"`, `"3xx"` 같은 카테고리가 나온다.

## Server Replay: 응답 녹화/재생

캡처된 응답을 저장해뒀다가 같은 요청이 오면 자동으로 재생한다.

### 매칭 로직

1. HTTP 메서드 정확 매칭
2. 전체 URL 정확 매칭
3. 둘 다 매칭되면 저장된 응답 반환 (서버 요청 없음)

### 사내 활용 사례

사내에서 이걸 가장 많이 쓴다. 백엔드 API가 아직 안 만들어졌거나, 개발 서버가 불안정할 때:

1. 한 번 정상 응답을 캡처
2. Server Replay에 추가
3. 이후 프론트엔드 개발은 프록시가 응답을 대신 반환

백엔드 의존성 없이 프론트엔드를 개발할 수 있다. 백엔드 API가 준비 안 됐다고 손 놓고 있을 필요가 없어졌다.

## HAR Export

트래픽을 HTTP Archive(HAR) 형식으로 내보낼 수 있다. 브라우저 DevTools에서도 쓰는 표준 포맷이라 호환성이 좋다.

버그 리포트할 때 HAR 파일을 첨부하면 재현 없이도 트래픽을 분석할 수 있어서 팀 내에서 유용하게 쓰고 있다.
