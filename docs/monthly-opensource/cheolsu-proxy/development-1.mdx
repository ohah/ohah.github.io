# 3-1. MITM 프록시 코어

## HTTPS 인터셉션의 핵심: CONNECT 터널링

브라우저가 HTTPS 사이트에 접속할 때, 프록시에게 먼저 `CONNECT` 요청을 보낸다:

```
CONNECT api.example.com:443 HTTP/1.1
Host: api.example.com:443
```

이 요청은 "나 `api.example.com`이랑 암호화 터널을 열고 싶으니까 중계해줘"라는 의미다. 일반 프록시는 이 터널을 그냥 통과시키지만, MITM 프록시는 여기서 가로챈다:

```
[브라우저] ─── TLS ──→ [프록시] ─── TLS ──→ [실제 서버]
                         ↑
                   여기서 복호화 후 재암호화
```

프록시가 하는 일:
1. `CONNECT` 요청을 받으면 `200 Connection Established` 응답
2. 클라이언트와의 TLS 핸드셰이크에서 **프록시의 CA로 서명한 가짜 인증서** 사용
3. 복호화된 평문 HTTP를 검사/조작
4. 실제 서버와는 별도의 TLS 연결로 통신

### 트러블슈팅: TLS ClientHello 파싱 <Badge type="danger">삽질</Badge>

초기에 모든 CONNECT 요청을 동일하게 처리했더니 레거시 클라이언트에서 TLS 핸드셰이크가 실패했다. 원인은 **TLS 버전 불일치**.

TLS 핸드셰이크의 첫 패킷인 ClientHello에는 클라이언트가 지원하는 TLS 버전 정보가 들어 있다. 이걸 파싱해서 클라이언트가 TLS 1.0/1.1만 지원하는지 확인해야 했다.

```
ClientHello 바이트 구조:
[0x16] [0x03, 0x01] [length] [handshake type] ... [client version]
 ↑ TLS   ↑ 레코드 버전              ↑ 실제 클라이언트 버전
```

이걸 바이트 레벨에서 파싱하는 `tls_version_detector.rs`를 만들었다. ClientHello를 읽어서:
- TLS 1.2+ → Rustls (현대적 TLS 라이브러리)
- TLS 1.0/1.1 → OpenSSL (레거시 지원)

이게 `hybrid_tls_handler` 모듈이다. 두 TLS 백엔드를 ClientHello 기반으로 자동 전환한다.

### 트러블슈팅: 인증서 캐싱 <Badge type="warning">성능</Badge>

도메인별로 인증서를 즉석 발급하는 건 비용이 크다. RSA 키 생성 + X.509 서명이 매번 일어나면 HTTPS 요청마다 수십 ms가 추가된다.

해결:
- `DashMap`(lock-free concurrent hashmap)으로 도메인별 인증서 캐시
- 한 번 발급한 인증서는 메모리에 보관
- 같은 도메인의 후속 요청은 캐시에서 즉시 반환

```rust
// DashMap을 쓰면 읽기가 lock-free라 동시 요청에서 병목이 없다
let cert_cache: DashMap<String, CertifiedKey> = DashMap::new();
```

## WebSocket 인터셉션

HTTP에서 WebSocket으로 업그레이드되는 과정도 가로채야 한다.

```
[클라이언트] → HTTP Upgrade 요청 → [프록시] → [서버]
[클라이언트] ← 101 Switching Protocols ← [프록시] ← [서버]
[클라이언트] ←→ WebSocket 프레임 ←→ [프록시] ←→ [서버]
```

`hyper-tungstenite`로 HTTP → WebSocket 업그레이드를 처리하고, `websocket_registry.rs`에서 활성 WebSocket 연결을 관리한다.

### 프로토콜 자동 감지

WebSocket 메시지 내용을 보고 프로토콜을 자동 감지한다:

- **Socket.IO (Engine.IO)** — 패킷 타입 접두사(0~5)로 감지. open/close/ping/pong/message 파싱
- **MQTT v3.1.1 / v5.0** — 고정 헤더의 패킷 타입 필드로 감지. CONNECT/PUBLISH/SUBSCRIBE 등 파싱

단순 텍스트/바이너리가 아닌 구조화된 형태로 보여줘서 디버깅이 훨씬 편하다.

### 메시지 주입

양방향으로 메시지를 주입할 수 있다:
- Client → Server 방향으로 가짜 메시지 전송
- Server → Client 방향으로 가짜 메시지 전송

테스트 시나리오를 만들거나, 서버의 특정 이벤트를 시뮬레이션할 때 유용하다.

## 인터셉트 규칙 엔진

5가지 작업 유형을 지원한다:

| 작업 | 설명 |
|------|------|
| **Block** | 요청 차단 (기본 403). 서버에 도달하지 않음 |
| **Modify Request** | 요청 헤더/바디 수정 후 전달 |
| **Modify Response** | 응답 상태/헤더/바디 수정 |
| **Map Local** | 로컬 파일로 응답 대체. 절대 경로 지정 |
| **Map Remote** | 다른 URL로 리다이렉트. 경로 보존 옵션 |

패턴 매칭은 와일드카드 기반이다:

```
*ads.example.com*       → 광고 도메인 차단
*api.example.com/v1/*   → 특정 API 경로 매칭
*.json                  → JSON 파일 매칭
```

### 트러블슈팅: 규칙 우선순위 충돌 <Badge type="danger">버그</Badge>

초기에 규칙 평가 순서가 일관되지 않아서, 같은 URL에 Block과 Modify 규칙이 모두 매칭되면 어떤 게 적용될지 예측할 수 없었다.

해결:
- 규칙에 명시적 우선순위(priority) 필드 추가
- 높은 우선순위가 먼저 평가
- 같은 우선순위면 먼저 등록된 규칙이 우선
- Block은 다른 규칙보다 항상 먼저 평가 (short-circuit)

## 네트워크 스로틀링

`throttle.rs`에서 대역폭 제한을 구현했다. 저속 네트워크 환경을 시뮬레이션할 수 있다.

토큰 버킷 알고리즘 기반으로, 설정된 대역폭만큼의 토큰을 주기적으로 채워넣고, 데이터를 전송할 때마다 토큰을 소비한다. 토큰이 부족하면 전송이 지연된다.

모바일 네트워크(3G/4G) 환경에서의 앱 동작을 테스트할 때 쓴다.

## 업스트림 프록시

회사 VPN이나 사내 프록시 뒤에서도 동작하도록 업스트림 프록시를 지원한다:

- HTTP/HTTPS 프록시
- SOCKS5 프록시
- 인증 (username/password)

```
[브라우저] → [철수 프록시] → [사내 프록시/VPN] → [인터넷]
```

사내에서 쓸 때 이게 필수였다. 회사 네트워크는 직접 외부 접속이 안 되니까.

### 트러블슈팅: SOCKS5 + TLS 조합 <Badge type="danger">삽질</Badge>

SOCKS5 업스트림 프록시를 통해 HTTPS 사이트에 접속할 때, SOCKS5 핸드셰이크 → TLS 핸드셰이크 순서가 꼬이는 문제가 있었다.

SOCKS5는 TCP 레벨 프록시라 TLS 핸드셰이크 전에 먼저 SOCKS5 연결을 수립해야 한다. 근데 Tokio의 비동기 특성상, SOCKS5 핸드셰이크가 완료되기 전에 TLS 핸드셰이크가 시작되는 경우가 있었다.

해결: SOCKS5 연결 수립을 `await`로 완전히 기다린 후에야 TLS 스트림을 감싸도록 순서를 강제했다. `socks5/` 모듈에서 연결 상태 머신을 명시적으로 관리한다.

:::info 1줄 요약
CONNECT 터널링 + 동적 인증서 발급으로 HTTPS를 인터셉트하고, Hybrid TLS Handler로 레거시 클라이언트까지 지원하며, WebSocket 프로토콜 자동 감지와 인터셉트 규칙 엔진으로 트래픽 조작의 기반을 잡았다.
