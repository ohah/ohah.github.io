# 3-9. 마무리

## CRD: 시작에서 완성까지

Chrome Remote Debugger(CRD)를 설계, 구현, 배포하기까지 개발 과정을 돌아보며 느낀 점들이 많았다. 시작은 '배민 따라하기'였지만, 진행하면서 점점 독립적인 특성과 문제 해결 과정이 가미되었다. 이번 마지막 파트에서는 CRD 프로젝트의 전반적인 마무리 과정과 앞으로 남은 과제를 정리하려고 한다.

## 3-tier 구조의 완성

초반 설계 당시 목표로 삼았던 3-tier 구조는 명확한 분리와 안정성을 기준으로 계속 다듬어졌다.

```
[웹 클라이언트] ←→ [WebSocket 중계 서버] ←→ [Inspector]
```

각 계층은 독립적으로 동작할 수 있도록 설정했고, 배포 후에도 유지보수가 용이하게 설계되었다. 특히 Inspector는 React 기반의 높은 구성 가능성을 최대한 활용했으며, 클라이언트는 Vite 기반 번들링으로 최적화를 진행했다.

- **웹 클라이언트의 주요 기능**: CDP 메시지 생성 및 서버 통신 관리
- **WebSocket 서버**: 중계 로직과 OTA(Over-The-Air) 업데이트 지원
- **Inspector UI**: 개발자를 위한 직관적이고 유연한 디버깅 인터페이스 제공

이 모든 것을 통합적으로 운영하는 과정에서 많은 시행착오가 있었지만, 결과적으로 예측 가능한 환경에서 안정적으로 작동하도록 구성했다.

## 남은 과제 및 미래 계획

### 1. 성능 최적화

현재 CRD는 실시간 로그와 Redux DevTools, Storage Inspector 등을 지원하지만, 성능이 데이터 크기에 따라 드롭되는 문제가 있다. 앞으로 IndexedDB 캐시 구조를 활용해 더 나은 데이터 로딩 성능을 제공할 계획이다.

### 2. 기능 확장

기존 Chrome DevTools와의 완벽한 호환성을 따라가기에는 아직 기능적인 갭이 존재한다. 특히, Profiler와 Performance 탭 같은 고급 분석 도구는 아직 구현되지 않았다. 차기 버전에서는 이러한 기능을 확장하는 데 주력할 것이다.

### 3. 커뮤니티 피드백

CRD는 배포 초기부터 커뮤니티의 많은 피드백을 받아왔다. 앞으로도 GitHub 이슈를 통해 개선 사항을 적극적으로 반영하고, 기여자를 위한 가이드를 더욱 명확히 다듬을 예정이다.

:::info 1줄 요약
CRD 개발의 마무리 단계에서는 안정성과 확장성을 모두 고려한 설계가 중심이었고, 성능 최적화와 추가 기능 개발이 향후 주요 과제로 남아 있다.
